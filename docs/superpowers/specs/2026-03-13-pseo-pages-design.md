# Mixler Programmatic SEO Pages — Design Spec

**Date:** 2026-03-13
**Status:** Approved

---

## Overview

Build ~96 programmatic SEO pages for mixler.ca across two page types: activity pages and guide pages. All content generated by Claude in session (no external API). Pages rendered as static HTML via Python/Jinja2 generator, with live Supabase event data injected client-side at page load.

**Goal:** Capture Calgary adult social events search traffic, funnel visitors to Mixler ticket purchases or waitlist signups, and use traffic data to inform future event planning.

---

## Architecture

### Data Layer

Content is stored as JSON files in `mixler-site/seo/data/`:

```
seo/
  data/
    activities/     # ~71 JSON files, one per activity
    guides/         # ~25 JSON files, one per guide
  templates/
    activity.html   # Jinja2 template
    guide.html      # Jinja2 template
  generate.py       # Renderer: JSON + templates → HTML pages
```

Generated pages output to:
```
mixler-site/
  activities/
    painting-calgary/index.html
    trivia-night-calgary/index.html
    ...
  guides/
    how-to-meet-people-calgary/index.html
    best-adult-social-events-calgary/index.html
    ...
```

### Generation Flow

1. `generate.py` reads all JSON files from `seo/data/`
2. Renders each file through the appropriate Jinja2 template
3. Outputs static HTML to `activities/` or `guides/` directories
4. Creates/overwrites `sitemap.xml` and `robots.txt` from scratch
5. Deploy via rsync: `rsync -avz --delete ~/mixler-site/ root@198.71.51.250:/var/www/mixler.ca/ --exclude='.git' --exclude='supabase' --exclude='scripts' --exclude='CLAUDE.md' --exclude='.github' --exclude='og' --exclude='seo'`
   Note: `--exclude='seo'` is added to prevent raw JSON, templates, and generator scripts from being served publicly.

### Supabase Live Data Injection (Activity Pages Only)

Each activity page's static HTML contains a placeholder div positioned after the intro, before the "What to Expect" section:

```html
<div id="events-slot" data-category-slug="painting"></div>
```

On page load, a small JS snippet:

1. Reads `data-category-slug` attribute from `#events-slot`
2. Queries Supabase via PostgREST join (no schema migration needed -- `event_categories` has a `slug` column):
   ```
   GET /rest/v1/events
     ?select=*,event_categories!inner(slug)
     &event_categories.slug=eq.{category_slug}
     &status=eq.published
     &event_date=gte.{today}
     &order=event_date.asc
     &limit=3
   ```
3. If events found: renders event cards into `#events-slot`
4. If no events: renders waitlist CTA into `#events-slot`

The JSON field for this is `category_slug` (matches the `slug` column in `event_categories`). The `event_categories` table has RLS enabled -- a public SELECT policy must exist or be created for the join to work with the anon key.

The SEO-critical content (title, meta, headings, intro, what to expect, tips, FAQ) is baked into the static HTML and fully crawlable. The event slot is non-critical for SEO.

### Waitlist Signup

When an activity page has no current Mixler events, the `#events-slot` renders a waitlist CTA form (email input + submit). On submit:

1. **Supabase insert** -- row written to a new `activity_waitlist` table: `(id, email, activity_slug, activity_name, created_at)`. Used for analytics: query which activities have the most demand.
2. **MailerLite group subscribe** -- user subscribed to a per-activity MailerLite group (e.g. "Waitlist: Axe Throwing"). When a new event is added for that activity, the group gets an announcement email.

Both calls are handled by a new Supabase Edge Function: `POST /functions/v1/join-activity-waitlist` with body `{ email, activity_slug, activity_name }`. The edge function uses the service role key internally, so the anon key is never exposed to the browser for writes.

**New Supabase migration required:** Create `activity_waitlist` table with a public INSERT policy (anon can write via the edge function, not directly).

**New MailerLite groups:** One group per activity, created on first signup if it doesn't exist (MailerLite API: `POST /api/groups` then `POST /api/subscribers`). Group naming convention: `Waitlist: [Activity Name]` (e.g. "Waitlist: Axe Throwing Calgary").

## Page Types

### 1. Activity Pages

**URL pattern:** `/activities/[activity-slug]/`
**Example:** `/activities/painting-calgary/`

**Page sections (all static except events slot):**

| Section | Content | SEO value |
|---|---|---|
| `<title>` + meta description | Deterministic template, not AI-generated | Primary keyword targeting |
| Breadcrumb | Home › Activities › [Name] | BreadcrumbList schema |
| Hero | Activity name, subtitle, category tags | H1 |
| Intro (2-3 sentences) | What the activity is, why popular in Calgary, who it's for | Body copy |
| Upcoming Mixler Events | **Live from Supabase.** Event cards or waitlist CTA | Conversion |
| What to Expect (4-6 items) | Structured grid: duration, skill level, group size, etc. | E-E-A-T signals |
| Tips (4-5 items) | Numbered list. Practical advice for first-timers | Dwell time |
| FAQ (4-6 Q&As) | Accordion. Rendered as FAQ schema markup | Rich result eligibility |
| Related Activities (sidebar) | 4 linked activity pages | Internal linking |
| Waitlist CTA (sidebar) | Pink card, white button | Conversion |

**Title template:** `[Activity Name] in Calgary | Mixler`
Example: `Painting Events in Calgary | Mixler`

**JSON schema:**

### 2. Guide Pages

**URL pattern:** `/guides/[guide-slug]/`
**Example:** `/guides/how-to-meet-people-calgary/`

**Two subtypes:**

- **Authority** (`"type": "authority"`): Mixler's voice, first-person, positions Mixler as the Calgary social expert. Stronger brand signal.
- **Roundup** (`"type": "roundup"`): Neutral resource listing options in Calgary. Mixler featured among them. Higher Google trust for competitive keywords.

**Page sections (all static):**

| Section | Content |
|---|---|
| `<title>` + meta | Deterministic template |
| Breadcrumb | Home › Guides › [Title] |
| Hero | Dark navy background, pink left stripe, title, byline, read time |
| Intro (pull quote style) | Pink left-border blockquote paragraph |
| Table of Contents (sidebar) | Numbered links to H2 sections |
| Content sections (3-5) | H2 + paragraphs + optional tip box |
| Inline Mixler CTA | Blue gradient block mid-article. Authority guides: prominent. Roundup guides: natural mention. |
| Related Activities grid | 4 linked activity pages |
| FAQ (4-6 Q&As) | FAQ schema markup |
| Sidebar CTA | Pink card, white/pink button |

**Title template:** `[Guide Title] | Mixler`

## SEO Elements (Every Page)

- Unique `<title>` tag (deterministic template, not AI-generated)
- Unique meta description (from JSON, written with target keyword)
- Canonical URL
- OG tags (title, description, image, URL) -- `og_image` baked into static HTML by `generate.py`, defaults to `images/mixler-logo-wide-color.png`
- BreadcrumbList schema
- FAQ schema (JSON-LD)
- Event schema on activity pages when live events are present (injected via JS)
- `sitemap.xml` entry

**Note on sitemap and robots.txt:** Neither `sitemap.xml` nor `robots.txt` currently exist in the repo. `generate.py` creates `sitemap.xml` from scratch on first run (not append). A `robots.txt` pointing to the sitemap is created as part of this project.

## Content Generation Process

All JSON content files are written by Claude directly in session (no external API calls needed). Process:

1. Generate all ~71 activity JSON files in session
2. Generate all ~25 guide JSON files in session
3. Build Jinja2 templates matching approved mockups
4. Build `generate.py` renderer
5. Run `generate.py` -- produces all HTML pages, creates `sitemap.xml` and `robots.txt` from scratch
6. Deploy via rsync
8. Regenerate OG previews

## Out of Scope (Phase 1)

- Neighbourhood variants (e.g. "painting events in Kensington") -- phase 2, data-driven
- Seasonal variants (e.g. "winter activities Calgary") -- covered by guide pages only
- Interactive filtering on activity pages -- phase 2
- Analytics tracking per page -- use existing setup
